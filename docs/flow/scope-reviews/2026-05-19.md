## Epics worth reconsidering
- **Severity-based epics (E1..E4) are a smell.** They forced E2-006 ↔ E4-005 to live in different epics despite being the same problem; same for E1-002 ↔ E4-004. Recommend re-cutting v0.8 by **story family**: `cli`, `caveman-scope`, `release-pipeline`, `observability`, `docs`, `tests`. Current severity tags can move into a `priority:` field on each story.
- **E5 FOLLOWUP** is currently a dumping ground for "stuff that fell out of v0.7" — same anti-pattern as a "Misc" epic. After merging E5-001/2/3 into one story and adding E5-005..008, E5 becomes a coherent **CLI + release-hardening** epic. Rename accordingly.
- **E1 BLOCKING** is effectively closed (only E1-002 npm-publish remains). After v0.8 cut, retire E1 and move npm-publish into a `release` epic alongside E4-002, E4-004.

## Risks of NOT trimming
If v0.8 ships against the current 5-epic / 32-story shape, the backlog will mis-attribute work: E2-006 and E4-005 will both get "closed_in" by the same upstream PR merge, creating duplicate ledger noise; E5-001..003 will close in a single commit that already shipped (post-hoc bookkeeping that no contributor wrote against an open story); E3-005 will sit "backlog" forever because dogfooding is a recurring activity, not a story. The npm-publish gate (E1-002) will continue to be cited as blocking everything in E1 even though E1-001/003 already shipped and the CLI works without npm. Net effect: sprint.yaml stops being a planning tool and becomes a retrospective log. That undermines the whole `/flow-sprint` premise.

## Honest read on v0.7

The release landed cleanly on the headline metric: 25 of 28 v0.6.1 review items closed in one day, with two real commits (7832ceb + e73f4bb) doing the heavy lifting and a third (0efbda0) adding bonus CLI parity. CI is green, schemas validate the shipped catalog, the doctor skill exists, docs cover the four surfaces a new user actually hits, and the migration path has real backup/rollback instead of aspirational prose. For a day-old project at v0.7, that is a credible posture. The npm-publish gate (E1-002) being deferred is the right call — the CLI works via `node bin/flow.js` and the slash commands are the primary surface, so blocking on a registry namespace would have been ceremony.

The surprises were mostly self-inflicted scope. E5 — "follow-up out-of-list improvements" — accumulated four stories of bonus CLI work (doctor, install, add, remove) that were not on the original 28-issue list. That is scope creep dressed up as completionism: the CLI parity is genuinely useful, but it should have been a v0.8 epic with its own planning pass, not bolted onto the v0.7 release commit. Similarly, E3-007 collapsed quickstart + profiles + adapters + migrate-from-bmad into a single "docs" story — closing it in 7832ceb attributes ~800 lines of docs to one ledger entry, which makes the changelog look denser than the underlying discipline warrants. The "HIGH" severity prefix on E2-006 (Caveman global scope) was over-priced: the actual fix is a one-line marker file plus a wait on upstream PR #407, and Flow ships fine without it. Conversely, E4-007 (README FAQ) was under-priced as "LOW" — it's the single most likely surface a curious user reads first, and the curl-pipe-bash and uninstall answers there are load-bearing trust signals.

Where discipline held: the severity-label preservation fix (E2-003), the double-compression fix (E3-006), the anchored marker regex (E3-001), and the auto-merge polling cap (E3-004) are all small, surgical, and exactly the kind of correctness work that justifies a 0.x → 0.7 jump rather than a 0.6.2. The decision to ship `tools/fix-caveman-shrink.sh` as print-only instead of auto-running `claude mcp` commands is the right kind of paranoia — modifying MCP registration is a session-wide side effect and Flow correctly refused to take that liability. The honest postmortem question for v0.8 is whether sprint.yaml should be re-cut by story family before the next batch of work starts, or whether one more cycle of severity-bucketed epics is fine. The evidence above says re-cut now: two of the four remaining stories already cross-reference each other across epic boundaries, which is the load-bearing symptom that the current epic axis is wrong.
